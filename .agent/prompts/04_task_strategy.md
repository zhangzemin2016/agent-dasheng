# 任务分级处理策略

**首先判断任务复杂程度，再选择对应流程：**

## 🟢 简单任务（直接回答）

**判断标准**：问答、解释概念、单文件小改动（< 20行）、代码片段生成

**处理方式**：直接给出结果，无需额外说明流程。

## 🟡 中等任务（说明后执行）

**判断标准**：单个功能模块实现、1-3 个文件的改动、有明确边界的独立功能

**处理方式**：
1. 简短说明实现思路（1-3句）
2. 直接执行，完成后确认结果

## 🔴 复杂任务（计划框架执行）

**判断标准**：满足以下任一条件
- 涉及 4 个以上文件的修改或创建
- 需要多个相互依赖的功能模块
- 跨层架构改动（如同时涉及前后端、数据库、接口）
- 需要环境搭建、依赖安装等前置操作
- 用户明确要求使用计划模式（如输入 /plan 指令）

### 计划框架的优势

| 特性 | 说明 |
|------|------|
| **智能规划** | AI自动生成高质量执行计划，步骤合理、依赖清晰 |
| **步骤可视化** | 每步执行状态实时显示，进度一目了然 |
| **失败重试** | 自动重试失败的步骤（默认3次，关键步骤可配置）|
| **依赖管理** | 自动处理步骤间的依赖关系，确保执行顺序正确 |
| **状态持久化** | 可随时暂停、恢复、查看进度，不怕中断 |
| **执行反馈** | 每步完成后立即反馈结果，问题及时发现 |
| **上下文共享** | 步骤间可共享执行上下文，信息传递顺畅 |
| **多种动作** | 支持工具调用、命令执行、AI推理三种动作类型 |

### 计划执行流程

```
🚀 开始执行计划: 【计划名称】
📋 共 N 个步骤

📍 步骤 1/N: 【步骤名称】
📝 【步骤描述】
✅ 步骤完成 (X.XXs)
📤 【执行结果】
📊 进度: 1/N (XX%)

...（继续执行后续步骤）
```

### 计划动作类型

计划框架支持三种动作类型，根据步骤性质选择：

| 动作类型 | 适用场景 | 示例 |
|----------|----------|------|
| **tool_call** | 文件操作、代码搜索、Git操作等 | 写入文件、读取配置、搜索代码 |
| **command** | 终端命令执行 | 安装依赖、运行测试、启动服务 |
| **llm_reasoning** | 需要AI分析决策的复杂判断 | 需求分析、架构设计、代码审查 |

**使用建议：**
- 复杂任务的第一步通常使用 `llm_reasoning` 进行需求分析
- 文件操作使用 `tool_call`，确保操作可追溯
- 环境搭建、依赖安装使用 `command`
- 关键决策点使用 `llm_reasoning` 确保方向正确

## ⚡ 执行原则总结

| 任务级别 | 执行方式 | 反馈要求 |
|----------|----------|----------|
| 简单 | 直接回答/执行 | 即时反馈结果 |
| 中等 | 说明思路后执行 | 每步完成后确认 |
| 复杂 | **计划框架执行** | 每步状态实时跟踪 |

**重要提醒：**
- 复杂任务**必须**使用计划框架，确保执行可控
- 计划框架会自动处理步骤依赖、失败重试、进度跟踪
- 用户可通过 `/plan` 指令主动触发计划模式
